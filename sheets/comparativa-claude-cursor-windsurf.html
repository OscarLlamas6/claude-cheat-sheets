01 / Conceptos clave

Tres carpetas que parecen lo mismo
pero hacen cosas distintas.

Antes de comparar, conviene aclarar el malentendido más común: aunque las tres carpetas viven en la raíz de tu proyecto y todas configuran un agente de AI, su modelo mental es radicalmente diferente.

.claude/ — Anthropic

Una mini-plataforma

Es la más rica de las tres. Trata al directorio como un sistema con cinco subsistemas reales: CLAUDE.md (memoria), rules/ (instrucciones modulares), skills/ (workflows reutilizables), agents/ (sub-agentes especializados) y settings.json (permisos + hooks). Pensada para Claude Code (terminal).

.cursor/ — Cursor

Reglas modulares con scope

Foco casi exclusivo en reglas. La carpeta principal es .cursor/rules/ con archivos .mdc (Markdown + YAML frontmatter) que se activan con globs o decisión del modelo. También aloja .cursor/mcp.json. La filosofía: muchos archivos pequeños, no uno gigante.

.windsurf/ — Cognition / Codeium

Reglas + workflows ejecutables

Mezcla de las dos anteriores pero más liviana: .windsurf/rules/ para directrices persistentes (con modos always_on, manual, model_decision, glob) y .windsurf/workflows/ para procedimientos reutilizables que Cascade ejecuta con /nombre-workflow.

03 / Estructuras reales

No, las tres carpetas no siguen la misma estructura.

Esta es probablemente la pregunta más importante que te hiciste. Veamos lado a lado cómo se ve cada una en un proyecto bien configurado al 2026.

.claude/ — la más completa

your-project/
├── CLAUDE.md                  # memoria del proyecto (commit)
├── CLAUDE.local.md             # overrides personales (gitignore)
└── .claude/
    ├── settings.json              # permisos + hooks (commit)
    ├── settings.local.json        # overrides locales (gitignore)
    ├── .mcp.json                  # servidores MCP del proyecto
    │
    ├── rules/                     # instrucciones modulares
    │   ├── code-style.md
    │   ├── testing.md
    │   └── api-conventions.md
    │
    ├── skills/                    # workflows auto-invocables
    │   ├── security-review/
    │   │   └── SKILL.md
    │   └── deploy/
    │       └── SKILL.md
    │
    ├── agents/                    # sub-agentes especializados
    │   ├── code-reviewer.md
    │   └── security-auditor.md
    │
    ├── hooks/                     # scripts deterministas
    │   └── block-secrets.sh
    │
    └── projects/                  # historial de sesiones (gitignore)

.cursor/ — minimalista, sólo reglas

your-project/
├── AGENTS.md                   # instrucciones cross-tool (recomendado 2026)
└── .cursor/
    ├── mcp.json                   # servidores MCP
    └── rules/                     # archivos .mdc con frontmatter
        ├── core.mdc               # always-on, <300 líneas
        ├── framework.mdc
        ├── architecture.mdc
        ├── testing.mdc
        └── security.mdc

# formato de un .mdc:
---
description: TypeScript conventions
globs: src/**/*.ts
alwaysApply: false
---
- Use strict TypeScript. No `any` types.
- Named exports only.

.windsurf/ — reglas + workflows

your-project/
├── AGENTS.md                   # instrucciones cross-tool (Cascade lo lee)
└── .windsurf/
    ├── rules/                     # reglas con activación inteligente
    │   ├── coding-standards.md    # always_on
    │   ├── react-patterns.md      # glob: *.tsx
    │   └── deployment.md          # manual (vía /deployment)
    │
    └── workflows/                 # procedimientos ejecutables
        ├── pre-pr-check.md        # /pre-pr-check
        └── release.md             # /release

# global del usuario:
~/.codeium/windsurf/
    ├── memories/
    │   └── global_rules.md
    └── global_workflows/
Conclusión rápida

Las estructuras NO son iguales. Skills y sub-agentes son únicos de Claude Code. Workflows ejecutables son únicos de Windsurf. Reglas con globs YAML son la fortaleza de Cursor. Lo único que tienen en común es que las tres aceptan Markdown y todas pueden leer (o están adoptando) AGENTS.md.

04 / Local vs Global

El mismo nombre en dos lugares distintos.

Cada una de las tres herramientas distingue entre configuración del proyecto (commit a git, compartido con el equipo) y configuración personal/global (sólo tuya, en tu home). La precedencia siempre es: proyecto > global.

Buena práctica

Lo que es convención de equipo va al proyecto y se commitea. Lo que es preferencia personal (tu estilo de comentarios, tus atajos, tus credenciales) va al global o al *.local.*. Mezclar los dos en el mismo archivo es la causa #1 de "el AI ignora mis reglas" en equipos compartidos.

05 / Equipos mixtos

Si en el equipo algunos usan uno y otros otro.

Este es el escenario real de la mayoría de los equipos en 2026. La respuesta corta: no necesitás tener las tres carpetas obligatoriamente, pero si querés que cada dev use su herramienta favorita, tenés tres caminos.

Camino 1 — AGENTS.md como única fuente

Es el camino más limpio y el que está ganando tracción en 2026. AGENTS.md es un estándar abierto promovido por Linux Foundation, adoptado por Codex CLI, Cursor, Windsurf, Copilot, Amp, Devin, Aider, Zed, Warp, RooCode y otros. Más de 40.000 proyectos open source ya lo usan.

  • Cursor lo lee nativamente
  • Windsurf lo lee nativamente (incluso por subdirectorio)
  • Claude Code aún no lo lee nativamente, pero podés crear un symlink: ln -s AGENTS.md CLAUDE.md
  • Un solo archivo Markdown, sin schema rígido

Camino 2 — Una fuente, varios outputs

Mantenés un archivo canónico (por ejemplo AGENTS.md o docs/ai-rules.md) y usás una herramienta open source que genera automáticamente los archivos específicos de cada IDE en cada commit.

  • Ideal cuando el equipo necesita reglas con scoping fino (globs en Cursor)
  • Permite features tool-specific donde aportan (skills de Claude, workflows de Windsurf)
  • Requiere disciplina: un script en CI o pre-commit que sincronice
  • Ver sección 06 para las herramientas concretas

Camino 3 — Convivencia explícita

Las tres carpetas conviven en el repo. Cada herramienta lee la suya y punto. Es lo que ves en la mayoría de los repos públicos hoy. Funciona, pero el riesgo es drift: alguien actualiza CLAUDE.md pero olvida .cursor/rules/ y de pronto un dev recibe sugerencias desactualizadas.

  • Setup más rápido inicialmente
  • Cada IDE aprovecha sus features únicos al 100%
  • Costo: mantenimiento manual o un script de sync casero
  • Recomendado cuando sólo 2 personas en el equipo usan IDE distinto al estándar

Lo que NO recomiendo

Crear las tres carpetas vacías "por las dudas". Cada una tiene su costo de mantenimiento y agrega ruido al repo. Los archivos sin contenido o con contenido aspiracional ("escribe código limpio") no aportan nada y le enseñan al equipo a ignorarlos.

  • Empezá con 1 archivo: AGENTS.md
  • Agregá tool-specific cuando un dev concreto lo pida con justificación
  • Versioná todo lo que sea de equipo; ignorá todo lo personal
  • Si nadie en el equipo usa Windsurf, no tengas .windsurf/
06 / Tooling open source

Herramientas que resuelven la fragmentación.

A mayo de 2026 hay un ecosistema en crecimiento de herramientas open source pensadas exactamente para tu problema. Estas son las que están vivas y vale la pena evaluar.

rule-porter

npx rule-porter --to claude-md

Conversor bidireccional entre Cursor (.mdc), Claude Code (CLAUDE.md), Copilot, Windsurf y AGENTS.md. Cero dependencias. Permite migrar legacy .cursorrules al nuevo formato .mdc y avisa cuando hay features que no se pueden traducir limpiamente (como globs en formatos planos). Ideal para migraciones one-shot.

rulebook-ai

rulebook-ai project sync --assistant cursor copilot

CLI más ambicioso: trabaja con "Packs" reutilizables (rules + memory + tools) que se despliegan al formato correcto de cada asistente. Soporta Cursor, Windsurf, Cline, RooCode, Kilo Code, Warp, Copilot, Claude Code, Codex CLI y Gemini CLI. Pensado para equipos que quieren un setup canónico versionable.

OpenSkills

npx openskills install anthropics/skills

Lleva el sistema de Skills de Anthropic a cualquier agente que pueda leer AGENTS.md: Cursor, Windsurf, Aider, Codex, Cline. Mismo formato SKILL.md que Claude Code. Con la flag --universal instala en .agent/skills/ en vez de .claude/skills/ para evitar conflictos.

OpenCode

112K+ ⭐ en GitHub · MIT License

Si querés evitar el problema de raíz, OpenCode es una alternativa terminal-native completamente open source. Soporta 75+ proveedores de LLM (Claude, GPT-5, Gemini, Ollama). Usa su propio formato pero también lee AGENTS.md. Útil si tu equipo prefiere consolidarse en una sola herramienta abierta.

Mi recomendación práctica

Si recién empezás: instalá rule-porter y mantené AGENTS.md como fuente única, generando los específicos al hacer commit (un hook de pre-commit lo hace en segundos). Si tu equipo ya tiene muchas reglas dispersas y querés un setup más estructurado: rulebook-ai con un Pack propio. Si necesitás Skills cross-IDE: OpenSkills.

07 / Estructura recomendada

La estructura que recomiendo en 2026.

Después de revisar lo que están haciendo equipos serios y las nuevas convenciones adoptadas en este último ciclo, esta es la estructura que minimiza fricción y maximiza compatibilidad cross-tool. 

your-project/
├── README.md                       # para humanos
├── AGENTS.md                       # ★ fuente única para AIs (cross-tool)
│
├── CLAUDE.md                       # symlink → AGENTS.md (o tool-specific real si necesitás)
├── .claude/                        # SOLO si usás Claude Code y necesitás:
│   ├── agents/                    #  · sub-agentes (no portables)
│   ├── skills/                    #  · skills (portables vía OpenSkills)
│   └── settings.json              #  · hooks y permisos
│
├── .cursor/                        # SOLO si necesitás scoping fino con globs:
│   └── rules/                     #  · generado desde AGENTS.md
│       ├── core.mdc               #    via rule-porter o rulebook-ai
│       └── frontend.mdc
│
├── .windsurf/                      # SOLO si querés workflows ejecutables:
│   ├── rules/                     #  · generado desde AGENTS.md
│   └── workflows/                 #  · únicos de Windsurf, valen la pena
│       ├── pre-pr-check.md
│       └── release.md
│
└── .gitignore                     # incluye: *.local.*, .claude/projects/

Reglas para que esto funcione bien

  • Una fuente de verdad: AGENTS.md en raíz. Si cambia algo de las convenciones del equipo, se cambia ahí primero.
  • Tool-specific sólo cuando aporta: sub-agentes y hooks de Claude, workflows de Windsurf, globs precisos de Cursor. Si no usás esa feature, no creés la carpeta.
  • Generación automática: un pre-commit hook que corre rule-porter o rulebook-ai project sync evita el drift entre archivos.
  • Personal vs equipo: todo lo del equipo va versionado. Settings personales en *.local.* o en el panel de la app, nunca en archivos commiteados.
  • CLAUDE.md como symlink: hasta que Claude Code soporte AGENTS.md nativamente, ln -s AGENTS.md CLAUDE.md en Linux/Mac (o copia + sync en Windows).
  • Empezá pequeño: 1 archivo AGENTS.md con tu stack, comandos de build/test, convenciones críticas y ban-list (carpetas que el AI no debe tocar). Iterá desde ahí.
08 / Respuestas directas

Tus preguntas en orden, sin vueltas.

P1¿Debo sí o sí tener las 3 carpetas?

No. Sólo debés tener la carpeta de la herramienta que tu equipo realmente usa. Crear las tres "por las dudas" es contraproducente: agrega ruido, mantenimiento y archivos vacíos que el AI puede leer mal. La regla simple: si nadie en el equipo usa Windsurf, no tengas .windsurf/.

P2¿Las tres carpetas siguen estructuras distintas?

Sí, completamente distintas. Lo viste en la sección 03: .claude/ tiene 5 subsistemas (agents, skills, rules, hooks, settings); .cursor/ básicamente tiene rules/ con archivos .mdc; .windsurf/ tiene rules/ y workflows/. Skills y sub-agentes son únicos de Claude. Workflows ejecutables son únicos de Windsurf. Las tres aceptan Markdown como formato base, pero ahí termina la similitud.

P3¿Qué pasa si en el equipo unos usan Cursor, otros Claude y otros Windsurf?

Tres opciones, en orden de preferencia: (1) Una sola AGENTS.md en raíz; las tres herramientas la leen (Claude vía symlink). (2) AGENTS.md como fuente y un sincronizador open source (rule-porter o rulebook-ai) que genera los archivos tool-specific en pre-commit. (3) Las tres carpetas conviviendo, mantenidas a mano: rápido al inicio, drift garantizado a los 3 meses.

P4¿Existen herramientas open source recientes que ayuden a esto?

Sí, varias activas en 2026: rule-porter (conversor bidireccional cero dependencias), rulebook-ai (CLI con sistema de Packs reutilizables que sincroniza a 10+ asistentes), OpenSkills (lleva el sistema de Skills de Anthropic a Cursor, Windsurf, Aider, Codex), y OpenCode (alternativa completa, MIT). Detalles en la sección 06.

P5¿Cuál es la mejor estructura? ¿Es la misma para los tres? ¿Puede ser una sola carpeta?

No es la misma. Cada herramienta tiene su estructura nativa y forzar una sola carpeta para los tres no funciona porque las features no son traducibles 1-a-1 (no podés tener un sub-agente persistente en Cursor, ni un workflow ejecutable con /comando en Claude Code, etc.).

Lo más cercano a "una sola carpeta" es tener un único AGENTS.md en raíz como fuente de verdad, y dejar que cada herramienta lea de ahí. Si necesitás features tool-specific, agregás esas carpetas sólo con lo que aporta valor (sub-agentes y hooks en .claude/, workflows en .windsurf/, reglas con scoping por glob en .cursor/rules/) y las generás automáticamente desde el AGENTS.md cuando se pueda. La estructura ideal está en la sección 07.

P6Mencionaste agents, skills, rules, commands... ¿es la misma estructura para los tres?

No. Esos nombres son del vocabulario de Claude Code y no existen igual en los otros dos:

Agents: únicos de Claude Code (.claude/agents/).
Skills: únicos de Claude Code (.claude/skills/SKILL.md); pueden replicarse en otros con OpenSkills.
Rules: existen en los tres pero con formatos distintos (Markdown plano en Claude, .mdc con frontmatter en Cursor, Markdown con modos de activación en Windsurf).
Commands: en Claude Code están deprecados y mergeados con Skills (2026); en Windsurf el equivalente son los Workflows; en Cursor no hay un equivalente directo (las reglas con description hacen algo parecido).

P7Resumen de un párrafo, por favor

Tres carpetas distintas para tres herramientas con filosofías distintas. No las tengas todas si no las usás todas. Para equipos mixtos: AGENTS.md en raíz como fuente única, complementada con carpetas tool-specific sólo donde agregan valor real (sub-agentes/hooks en .claude, workflows en .windsurf, reglas con globs en .cursor). Sincronizá con rule-porter o rulebook-ai para evitar drift. La estructura ideal completa está en la sección 07.
