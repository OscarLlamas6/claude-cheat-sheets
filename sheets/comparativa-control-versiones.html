Sistemas de control de versiones // cuadro comparativo

Ocho tecnologías de versionamiento de código, ordenadas por año de aparición. Se agrupan GitHub, GitLab, Bitbucket, CodeCommit, Forgejo y Gitea bajo Git, ya que todas implementan el mismo motor: son plataformas de alojamiento, no sistemas de control de versiones distintos.

¿Qué es? Sistema de control de versiones distribuido. Cada clon del repositorio es una copia completa con todo el historial. Sistema centralizado: existe un único servidor con la verdad; los clientes descargan solo la revisión que necesitan. Sistema distribuido, contemporáneo de Git, con filosofía muy similar pero interfaz de comandos más simple y consistente. Motor de control de versiones distribuido de nueva generación, compatible con repos Git existentes, pensado para reescribir historial sin miedo. Sistema centralizado orientado a archivos binarios grandes (assets, videos, planos CAD), muy usado en videojuegos y estudios de animación. Sistema distribuido que empaqueta control de versiones + wiki + issues + foro, todo en un único archivo SQLite. Sistema que puede operar centralizado o distribuido ("branch per task"), con foco en archivos grandes y ramas visuales. Uno de los primeros sistemas de control de versiones centralizados de uso amplio; versiona archivo por archivo, no el proyecto como conjunto.
¿Qué hace bien? Ramas y fusiones baratísimas, trabajo 100% offline, ecosistema enorme, estándar de facto de la industria. Control de acceso granular por carpeta, buen manejo de binarios grandes, historial lineal fácil de auditar. Comandos predecibles y coherentes, buen rendimiento en repos con historiales muy largos. Deshacer/rehacer cualquier operación, editar commits antiguos sin "rebase interactivo" doloroso, un solo modelo mental para todo. Bloqueo de archivos (locking) para que dos personas no pisen el mismo binario, control fino de permisos, escalabilidad a TB de datos. Cero dependencias externas: un solo ejecutable y un archivo para todo el proyecto, ideal para autocontención. Interfaz gráfica pulida, diffs visuales de assets de Unity/Unreal, mezcla de modelos centralizado y distribuido. Simplicidad histórica, funcionó como estándar durante los 90 y 2000.
¿Qué NO hace / limitaciones? No maneja bien archivos binarios grandes de forma nativa (necesita Git LFS aparte); curva de aprendizaje pronunciada. Sin servidor no hay nada que hacer; ramas y fusiones son costosas y propensas a conflictos. Ecosistema y adopción mucho menores que Git; GitHub eliminó soporte nativo de Hg hace años. Todavía joven: menos documentación, menos integraciones de terceros, comunidad pequeña comparada con Git. No es adecuado para código abierto ni proyectos pequeños; licenciamiento y administración de servidor complejos. Adopción marginal fuera del propio proyecto SQLite; pocas integraciones con CI/CD modernas. Menos estándar fuera de la industria de videojuegos; curva de aprendizaje del modelo mixto. Sin commits atómicos multi-archivo, ramas casi inutilizables, sin manejo real de renombrados. Prácticamente abandonado.

Cuándo usarlo Prácticamente cualquier proyecto de software moderno, solo o en equipo, open source o privado. Organizaciones con estructura jerárquica estricta que necesitan control de acceso por carpeta y ya tienen infraestructura SVN. Si el equipo ya lo domina o si se mantiene un proyecto histórico creado en Mercurial (ej. partes de Mozilla, Facebook internamente lo usó años). Desarrolladores que quieren la potencia de Git pero con una UX más limpia, o que reescriben historial constantemente. Estudios de videojuegos, animación o ingeniería con archivos binarios enormes y necesidad de bloqueo de archivos. Proyectos pequeños/personales que quieren versionado + wiki + tickets sin montar infraestructura adicional. Equipos de videojuegos que usan Unity/Unreal y necesitan diffs visuales de assets. Solo para mantener sistemas legados que aún corren sobre CVS. No para proyectos nuevos.
Cuándo NO usarlo Si el equipo maneja principalmente archivos binarios pesados sin configurar Git LFS. Proyectos open source distribuidos globalmente o equipos que necesitan trabajar offline. Si se necesita integración amplia con plataformas modernas (la mayoría ya no lo soporta de forma nativa). Si el equipo depende de integraciones maduras de CI/CD, code review o hosting que aún no soportan jj de forma nativa. Proyectos pequeños, open source, o con presupuesto limitado. Proyectos grandes con muchos colaboradores externos que ya esperan un flujo tipo GitHub/GitLab. Software genérico fuera del ámbito de videojuegos, o equipos con presupuesto ajustado. En cualquier proyecto nuevo: fue reemplazado por SVN y luego por Git.
