PyTorch 01 / 06

"El framework de los investigadores."

Qué es

Framework de deep learning de código abierto creado por Meta. Ofrece tensores con aceleración GPU y un sistema de autograd dinámico que permite construir redes neuronales como si fueran código Python normal.

Qué hace

Construye, entrena y despliega redes neuronales. Soporta CUDA, MPS (Apple Silicon) y aceleradores. Tiene ecosistema rico: TorchVision, TorchAudio, TorchServe, y es la base de Hugging Face Transformers.

Lo que hace que otros NO

Grafos computacionales dinámicos (define-by-run): puedes hacer if, bucles y debugging con print() dentro del modelo. Es el framework de la investigación moderna — la mayoría de papers nuevos publican código en PyTorch.

Lo que NO hace bien

No es ideal para ML clásico (no tiene RandomForest ni SVM). El despliegue en producción, aunque mejorado con TorchScript y ExecuTorch, históricamente fue menos pulido que TensorFlow.

Úsalo cuando
  • Investigas o experimentas con arquitecturas nuevas
  • Quieres debugging fácil y código pythónico
  • Usas Hugging Face o modelos open source
  • Necesitas control fino del training loop
Evítalo cuando
  • Tu problema es ML clásico tabular
  • El equipo no tiene experiencia en Python avanzado
  • Necesitas TFLite o despliegue web maduro
  • Solo quieres entrenar una regresión simple
TensorFlow 02 / 06

"El framework de los ingenieros de producción."

Qué es

Framework de deep learning de Google, diseñado desde el inicio pensando en escalar de la laptop al datacenter. Desde la versión 2.x usa Keras como API por defecto y soporta eager execution.

Qué hace

Construye y entrena redes neuronales con soporte para CPU, GPU y TPU nativo. Ofrece un ecosistema completo: TFX (pipelines), TF Serving (despliegue), TFLite (móvil/embedded), TF.js (navegador).

Lo que hace que otros NO

El ecosistema de producción más completo del mercado: del prototipo a un Android, navegador o Raspberry Pi sin cambiar de framework. Soporte de TPUs nativo y maduro.

Lo que NO hace bien

Ha perdido terreno frente a PyTorch en investigación académica. Su API histórica (TF 1.x) era notoriamente compleja, y aunque TF 2.x mejoró, arrastra deuda de complejidad. No hace ML clásico.

Úsalo cuando
  • Necesitas desplegar modelos a escala empresarial
  • Vas a TPUs de Google Cloud
  • El destino es móvil, web o IoT/edge
  • Tu equipo es de MLOps tanto como de ML
Evítalo cuando
  • Estás haciendo investigación experimental
  • Quieres seguir papers recientes (mayoría usan PyTorch)
  • Eres principiante absoluto en DL
  • Tu problema es ML tabular clásico
Scikit-learn 03 / 06

"La navaja suiza del ML clásico."

Qué es

Librería Python de machine learning clásico (no deep learning). Construida sobre NumPy, SciPy y matplotlib. Su API .fit() / .predict() se ha vuelto el estándar de facto en toda la industria.

Qué hace

Clasificación, regresión, clustering, reducción de dimensionalidad, selección de modelos, preprocesamiento, pipelines reproducibles, validación cruzada y métricas. Decenas de algoritmos clásicos perfectamente implementados.

Lo que hace que otros NO

La API más consistente y elegante del ecosistema ML: cualquier modelo se entrena igual. Documentación impecable. Es la primera herramienta que debe tocar cualquier data scientist antes de saltar a algo más complejo.

Lo que NO hace

No hace deep learning. No usa GPU de forma nativa. No escala bien a datasets de cientos de millones de filas. No hace online learning robusto. No procesa imágenes ni texto crudos sin extracción de features previa.

Úsalo cuando
  • Trabajas con datos tabulares estructurados
  • Necesitas un baseline rápido y honesto
  • Estás aprendiendo ML
  • El dataset cabe en memoria (hasta ~10M filas)
Evítalo cuando
  • Trabajas con imágenes, audio o texto crudo
  • Tu dataset es masivo (centenas de millones)
  • Necesitas redes neuronales
  • Quieres GPU para entrenar más rápido
JAX 04 / 06

"NumPy con esteroides — y un compilador."

Qué es

Librería de Google para computación numérica de alto rendimiento. Combina una API casi idéntica a NumPy con autodiferenciación, compilación JIT vía XLA, y paralelización automática en GPU/TPU.

Qué hace

Ofrece transformaciones componibles: grad (derivadas), jit (compila), vmap (vectoriza), pmap (paraleliza entre dispositivos). Sobre JAX se construyen frameworks como Flax, Haiku y Equinox.

Lo que hace que otros NO

Filosofía funcional pura: las transformaciones se componen como funciones matemáticas. Esto permite optimizaciones imposibles en otros frameworks. Es la herramienta interna de DeepMind para entrenar modelos masivos como Gemini.

Lo que NO hace bien

No es un framework "completo": para hacer DL necesitas Flax o Haiku encima. Tiene menos modelos pre-entrenados que PyTorch, ecosistema más pequeño y curva de aprendizaje empinada (efectos secundarios prohibidos, PRNG explícito).

Úsalo cuando
  • Haces investigación de vanguardia que necesita velocidad
  • Tu problema es científico (física, química, biología)
  • Necesitas paralelizar entre muchas TPUs
  • Te gusta la programación funcional
Evítalo cuando
  • Eres principiante en DL
  • Quieres usar modelos pre-entrenados de la comunidad
  • Necesitas despliegue tradicional en producción
  • Vienes de un equipo OOP sin cultura funcional
Keras 05 / 06

"Deep learning para humanos."

Qué es

API de alto nivel para construir redes neuronales con código mínimo. Desde Keras 3 es multi-backend: el mismo código corre sobre TensorFlow, PyTorch o JAX según prefieras.

Qué hace

Permite definir modelos capa por capa (Sequential) o con la API funcional. Ofrece model.fit(), callbacks, métricas, capas listas para usar (CNN, RNN, Transformer) y un workflow muy ergonómico.

Lo que hace que otros NO

Simplicidad radical: una red neuronal entera en 5 líneas. Y desde Keras 3, eres backend-agnostic: pruebas tu modelo en TF, PyTorch y JAX cambiando una variable de entorno.

Lo que NO hace bien

Esa simplicidad se paga con menos control: training loops totalmente personalizados, gradientes manuales, o arquitecturas muy exóticas se sienten incómodas. Las abstracciones a veces ocultan bugs sutiles.

Úsalo cuando
  • Estás aprendiendo deep learning
  • Quieres prototipar rápido (CNN, RNN, Transformer estándar)
  • Quieres flexibilidad de backend (TF/PyTorch/JAX)
  • Enseñas DL a otros
Evítalo cuando
  • Tu arquitectura es muy custom o exótica
  • Necesitas control total del training loop
  • Replicas papers con código complejo en PyTorch puro
  • Quieres optimizar cada microsegundo
XGBoost 06 / 06

"El campeón silencioso de los datos tabulares."

Qué es

Librería de gradient boosting sobre árboles de decisión, ultra-optimizada en C++ con bindings para Python, R, Java, Scala, etc. Hermana de LightGBM y CatBoost, que cumplen un rol similar.

Qué hace

Entrena ensambles de árboles con boosting iterativo. Maneja valores faltantes nativamente, ofrece regularización L1/L2, soporta GPU, y se integra con la API de Scikit-learn (.fit()/.predict()).

Lo que hace que otros NO

Es el rey absoluto de Kaggle en problemas tabulares: una y otra vez supera a redes neuronales en datos estructurados. Velocidad y precisión que ni siquiera Scikit-learn alcanza en gradient boosting clásico.

Lo que NO hace

No procesa imágenes, texto crudo ni audio: necesita features tabulares. No hace deep learning. Requiere feature engineering — no aprende representaciones por sí solo. Tuning de hiperparámetros puede ser tedioso.

Úsalo cuando
  • Tienes datos tabulares estructurados
  • Necesitas la mayor precisión posible
  • Compites en Kaggle o ganas dinero con scoring
  • Casos de fraude, riesgo crediticio, churn
Evítalo cuando
  • Trabajas con imágenes, audio, texto, video
  • Tu dataset es minúsculo (decenas de filas)
  • Necesitas extrapolación fuera del rango de entrenamiento
  • Quieres un modelo súper interpretable (usa regresión)

Decisión rápida

// Si pudieras hacer una sola pregunta, sería esta

"Mis datos son una tabla con columnas y filas."

Empieza con Scikit-learn para baseline. Si necesitas más precisión, salta a XGBoost. No toques redes neuronales hasta agotar estas opciones.

"Mi problema es imágenes, texto o audio."

Necesitas deep learning. Usa PyTorch si vienes de research o Hugging Face; TensorFlow si tu meta es producción móvil/edge; Keras si quieres simplicidad.

"Estoy aprendiendo desde cero."

Aprende Scikit-learn primero (ML clásico). Luego Keras para tu primera red neuronal. PyTorch puro y JAX vienen después, no antes.

"Voy a desplegar en móvil o navegador."

TensorFlow es claramente la opción adulta: TFLite para Android/iOS, TF.js para navegador. PyTorch tiene ExecuTorch pero está menos maduro.

"Hago research y quiero publicar papers."

PyTorch es el estándar de facto en investigación. Si vas por modelos masivos en TPU o quieres velocidad pura, considera JAX + Flax.

"Tengo poco tiempo y solo quiero un modelo decente."

Para tabular: XGBoost con hiperparámetros por defecto. Para visión/NLP: un modelo pre-entrenado de Hugging Face en PyTorch. No reinventes la rueda.
