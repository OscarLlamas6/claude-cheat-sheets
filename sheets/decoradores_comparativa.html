En detalle

— Qué hace cada uno por detrás

@staticmethod
Una función "normal" que vive dentro del namespace de la clase
¿Qué es?

Decorador integrado en Python. Marca un método que no recibe ni self ni cls. Se invoca directamente desde la clase: Clase.metodo(args), sin necesidad de tener una instancia.

Por detrás

staticmethod es un descriptor que envuelve a la función. Cuando Python accede al atributo, el descriptor devuelve la función original sin hacer ningún binding (sin inyectar self ni cls). Es prácticamente azúcar sintáctico para decir "esta función está agrupada con la clase pero es independiente de cualquier instancia".

Qué hace el ORM

Nada. SQLAlchemy ignora completamente los @staticmethod al construir el mapeo. No los inspecciona, no los toca — son invisibles para el ORM.

En tu código

expire_last_info(location). Lo invocas como StoragePools.expire_last_info(loc_id). Como no necesita self, podría perfectamente vivir como función fuera de la clase, pero está aquí para mantener la operación lógicamente agrupada con el modelo.

@staticmethod
def expire_last_info(location):
    sql_update = "UPDATE storage_pools SET last_info = False WHERE location_id = :location"
    db.session.execute(text(sql_update), {'location': location})
    db.session.commit()
@classmethod
Un método cuyo primer argumento es la clase, no la instancia
¿Qué es?

Decorador integrado en Python. El primer argumento del método es cls (la clase), no self. Útil para constructores alternativos (cls.from_dict(...)) o para queries que respeten herencia.

Por detrás

También es un descriptor. Al acceder al atributo (sea desde la clase o desde una instancia) devuelve un MethodType ya enlazado a la clase. Por eso siempre recibes cls sin importar cómo se llame al método.

Qué hace el ORM

Nada directo. SQLAlchemy no lo trata especial. Pero en patrones tipo Active Record es común usarlo así:

@classmethod
def active_in_location(cls, location_id):
    return db.session.query(cls).filter_by(
        location_id=location_id, last_info=True
    ).all()

La gracia es que cls se resuelve a la subclase real, así que si heredas el modelo el método sigue funcionando.

En tu código

No lo usas. Pero expire_last_info sería un candidato natural para reescribirlo como classmethod, así eliminas el SQL crudo y aprovechas el ORM:

@classmethod
def expire_last_info(cls, location):
    db.session.query(cls).filter_by(location_id=location)\
        .update({"last_info": False})
    db.session.commit()
@property
Atributo computado a nivel de instancia — invisible al ORM
¿Qué es?

Decorador estándar de Python. Convierte un método en un atributo de instancia computado. Lo accedes como obj.x (sin paréntesis), pero por debajo se ejecuta el getter.

Por detrás

property es un descriptor de datos. Intercepta el acceso al atributo, llama al getter pasándole self, y devuelve el valor calculado. Cada acceso re-ejecuta el getter (sin caché).

Qué hace el ORM

NADA — y esto es crítico. SQLAlchemy no sabe que existe available_tb. Si intentas usarlo en un query:

# ❌ NO funciona
session.query(StoragePools).filter(StoragePools.available_tb > 5)
session.query(StoragePools).order_by(StoragePools.used_tb)

Falla, porque StoragePools.available_tb a nivel de clase es un objeto property, no una Column ni una expresión SQL. El cálculo solo ocurre en Python, después de que SQLAlchemy haya cargado las filas y construido las instancias en memoria.

En tu código

available_tb, capacity_tb, provisioned_tb, used_tb, p_pct. Todos toman columnas en MB y las convierten a TB o porcentajes en Python puro. Perfectos para mostrar en plantillas Jinja, pero no se pueden usar en WHERE ni ORDER BY SQL.

@property
def available_tb(self):
    return round(self.available_mb / 1024 / 1024, 2)
@hybrid_property
Funciona en dos contextos: como property en Python y como expresión en SQL
¿Qué es?

Decorador de la extensión sqlalchemy.ext.hybrid. Crea un atributo que se comporta de dos formas distintas según desde dónde se acceda:

· Desde una instancia (obj.x) → ejecuta el cuerpo Python con self, igual que @property.
· Desde la clase (Modelo.x) → se interpreta como expresión SQL.

Por detrás

Es un descriptor "híbrido" más sofisticado que property. Detecta el tipo de acceso (instancia vs clase) y escoge entre dos caminos:

· Acceso por instancia → ejecuta el getter Python tal cual.
· Acceso por clase → toma el mismo cuerpo y lo trata como una expresión SQLAlchemy, asumiendo que los self.col referencian Columns. SQLAlchemy lo compila a SQL.

Ese auto-mapeo solo funciona si el cuerpo usa operadores que SQLAlchemy sabe traducir (sumas, comparaciones, and_, or_, etc.). Si haces algo demasiado "Python-y" — como acceder a un relationship (self.location.name) — la traducción automática falla y debes definir .expression aparte.

Qué hace el ORM

SQLAlchemy lo registra como atributo cualificado del mapeo. Cuando lo usas en filter, order_by, select, etc., genera el SQL correspondiente y lo embebe en la query — todo sin cargar instancias en memoria.

En tu código

sort_key. El getter Python usa self.location.name (cruza un relationship), por eso la versión SQL no se puede inferir automáticamente y necesitaste declarar @sort_key.expression aparte.

@hybrid_property
def sort_key(self):
    # Contexto Python: usa el relationship cargado
    location_name = self.location.name if self.location else ""
    return (location_name, self.storage_name, self.pool_name)
@sort_key.expression
La forma SQL del hybrid_property — se invoca cuando el acceso es a nivel de clase
¿Qué es?

El "complemento SQL" de un @hybrid_property. Define qué expresión SQL se genera cuando el atributo se usa a nivel de clase. Recibe cls y debe retornar una expresión SQLAlchemy compilable a SQL (Column, scalar_subquery, func.*, tuple_, etc.).

Por detrás

Sobrescribe el comportamiento por defecto del hybrid_property en contexto de clase. Cuando SQLAlchemy detecta acceso desde la clase (Modelo.sort_key, no instancia.sort_key), llama a este método con cls=Modelo y usa el ClauseElement que retornes como parte del SQL final.

Sin él, SQLAlchemy intentaría reusar el getter del @hybrid_property y, en tu caso, fallaría porque self.location.name no es traducible.

Qué hace el ORM

Embebe esa expresión SQL en el statement compilado. Cuando haces:

query.order_by(StoragePools.sort_key)

SQLAlchemy genera SQL aproximadamente así:

ORDER BY
  (SELECT location.name FROM location
   WHERE location.id = storage_pools.location_id),
  storage_pools.storage_name,
  storage_pools.pool_name

Es decir: una subquery correlacionada para traer location.name + un ORDER BY multicolumna. Todo en una sola query SQL, sin cargar instancias en memoria, y haciendo el sort en la base de datos.

En tu código

La definición construye la subquery con select(...).correlate(cls).scalar_subquery() y luego usa tuple_(...) para el sort multicolumna:

@sort_key.expression
def sort_key(cls):
    location_name_sub = (
        select(Location.name)
        .where(Location.id == cls.location_id)
        .correlate(cls)
        .scalar_subquery()
    )
    return tuple_(location_name_sub, cls.storage_name, cls.pool_name)
@declared_attr
Declaración diferida de columnas/relaciones — clave para mixins reutilizables
¿Qué es?

Decorador de SQLAlchemy (sqlalchemy.orm.declared_attr). Permite declarar columnas, relaciones u otros atributos del mapeo de forma diferida: la función no se ejecuta al definir la clase, sino una vez por cada clase mapeada que la herede.

Por detrás

La función decorada queda como un placeholder en la clase. Cuando SQLAlchemy construye el mapeo de cada modelo concreto, invoca esa función pasándole cls (la subclase real) y usa lo que retorne como el atributo definitivo (un Column, un relationship, etc.).

Esto es importante porque cada subclase obtiene su propia instancia de la columna, no una compartida — algo que sería un problema si usaras simplemente created_at = Column(...) en un mixin.

Qué hace el ORM

Es la herramienta canónica para mixins de mapeo reutilizables. Permite construir columnas dinámicamente usando cls.__name__, cls.__tablename__, etc. — útil para columnas como created_by, tenant_id, o claves foráneas que dependen del nombre de la tabla.

En tu código

No lo usas. Pero sería útil si quisieras un mixin de auditoría reutilizable para todos tus modelos:

class TimestampMixin:
    @declared_attr
    def created_at(cls):
        return Column(DateTime, default=datetime.utcnow)

    @declared_attr
    def updated_by(cls):
        return Column(String(50))


class StoragePools(TimestampMixin, Model):
    # StoragePools recibe sus propias columnas created_at y updated_by
    ...

Cada modelo que herede de TimestampMixin recibe sus propias columnas como si las hubieras escrito a mano.

⚡ Para llevarte

Las decoradores de Python (@staticmethod, @classmethod, @property) son invisibles para el ORM. Solo existen en Python, después de cargar las filas.

Las decoradores de SQLAlchemy (@hybrid_property, .expression, @declared_attr) son las que el ORM sí entiende: las traduce a SQL para que puedas filtrar, ordenar y declarar columnas dinámicas en la base de datos.

En tu modelo, los @property sirven para mostrar TB en plantillas, pero no podrías hacer filter(StoragePools.used_tb > 100). Para eso, tendrías que migrar esos cálculos a @hybrid_property.
