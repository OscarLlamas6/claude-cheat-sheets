KV Cache — Cuantificando el costo de memoria

Durante inferencia, el transformer guarda las matrices Key y Value de cada token generado para no recalcularlas. Este cache crece linealmente con el contexto y puede dominar el uso de VRAM. La siguiente fórmula y visualización usan los parámetros reales de Llama 3.3 70B (Unsloth).

Fórmula — KV Cache Bytes

Glosario visual — cada término de la fórmula
Factor K & V
× 2
El mecanismo de atención necesita dos matrices por capa: Keys (calcula cuánto atender a cada token) y Values (qué información extraer). Mismo tamaño, siempre van juntas.
Batch size
× 1
Secuencias procesadas en paralelo. Cada elemento del batch tiene su propio KV cache completo e independiente. La memoria escala linealmente con el batch.
n_hidden_layers
80 capas
El modelo apila 80 capas transformer. Cada capa mantiene su propio par de matrices K y V. A más profundidad, mayor capacidad de razonamiento y mayor costo de memoria.
n_kv_heads — GQA
8 de 64
Grouped Query Attention: 64 Q heads pero solo 8 KV heads. Cada grupo de 8 queries comparte 1 par K/V. Ahorro real: ~8× menos memoria vs Multi-Head Attention estándar.
head_dim
128 dims
Tamaño del vector K o V por cada head. Se calcula como hidden_size / num_heads = 8192 / 64 = 128. Cada K y V es un vector de 128 floats por head.
seq_len (contexto)
8,192
Tokens en el contexto activo. El cache crece con cada token generado. Llama 3.3 soporta hasta 131,072 tokens con RoPE escalado (factor ×8 sobre los 8,192 originales del pretraining).
bytes_per_param
2 bytes
bfloat16: 16 bits (1 signo, 8 exponente, 7 mantisa). Mayor rango dinámico que float16 con la misma memoria. Es el torch_dtype del config de Llama 3.3.
