01
Prompt Engineering
Hablarle bien al modelo para obtener lo mejor
$ Bajo Fácil
▾
📖 ¿Qué es?

Técnica de diseño y optimización de los inputs (prompts) que se le dan al modelo. No modifica el modelo en sí; modifica cómo se le formula la pregunta o instrucción. Incluye técnicas como Chain-of-Thought (CoT), Few-shot, Zero-shot, ReAct, Tree of Thoughts, etc.

⚙️ ¿Qué hace?

Guía al modelo paso a paso, proporciona ejemplos, descompone problemas complejos, establece roles o personas, y delimita el formato de salida esperado. Con un buen prompt se pueden extraer capacidades latentes del modelo sin costosos entrenamientos.

✅ Ventajas
  • Sin costo de entrenamiento
  • Inmediato, iterable en segundos
  • Portable entre modelos
  • No requiere datos propios
  • Ideal para prototipos rápidos
❌ Desventajas
  • Frágil: un token cambia el output
  • Ocupa ventana de contexto
  • No persiste entre sesiones
  • Límite de lo que el modelo "sabe"
  • Difícil de versionar y testear a escala
📊 Métricas clave
  • Tokens usados por prompt
  • Consistencia de salida
  • Latencia de respuesta
  • Pass rate en test suite
🟢 Cuándo USARLO
  • Prototipado y exploración rápida
  • Tareas bien definidas y acotadas
  • Sin datos propios de entrenamiento
  • Presupuesto casi cero
  • Querés usar modelos de API existentes
🔴 Cuándo NO usarlo
  • Requiere conocimiento muy especializado o propietario
  • Consistencia crítica en producción masiva
  • Necesitás outputs estructurados 100% confiables
  • El modelo no tiene el conocimiento base necesario

02
RAG — Retrieval-Augmented Generation
Dale al modelo una biblioteca en tiempo real
$$ Medio Intermedio
▾
📖 ¿Qué es?

Arquitectura que combina un sistema de recuperación de información (búsqueda en bases de datos vectoriales, BM25, o híbrida) con un LLM generador. Antes de generar, el sistema recupera documentos relevantes y los inyecta en el contexto del modelo, permitiéndole "acceder" a información que no estaba en su entrenamiento.

⚙️ ¿Qué hace?

Convierte documentos en embeddings vectoriales almacenados en una base de datos vectorial (Pinecone, Weaviate, Chroma, etc.). En tiempo de inferencia, la consulta del usuario se convierte también en embedding y se buscan los K documentos más similares, que se pasan como contexto al LLM.

✅ Ventajas
  • Conocimiento actualizable sin re-entrenar
  • Citable y verificable (fuentes)
  • Funciona con cualquier LLM
  • Escalable a millones de docs
  • Reduce alucinaciones en dominio
❌ Desventajas
  • Latencia adicional por búsqueda
  • Calidad depende del chunking
  • Costoso con grandes volúmenes
  • Falla en razonamiento implícito complejo
  • Embeddings se desactualizan
🏗️ Variantes
  • Naive RAG (básico)
  • Advanced RAG (re-ranking)
  • Modular RAG
  • GraphRAG (relaciones)
  • HyDE (hipotético)
  • Self-RAG (auto-evaluación)
🟢 Cuándo USARLO
  • Documentación empresarial / legal / técnica
  • Knowledge base de atención al cliente
  • Información que cambia frecuentemente
  • Necesitás transparencia (fuentes citables)
  • Datos propietarios o confidenciales
🔴 Cuándo NO usarlo
  • Razonamiento puro sin consulta documental
  • Cambios de comportamiento/tono del modelo
  • Datos muy pequeños (pocas páginas)
  • Latencia ultra-baja es crítica

03
Fine-Tuning
Re-entrenar el modelo con tus propios datos
$$$ Alto Avanzado
▾
📖 ¿Qué es?

Proceso de continuar el entrenamiento de un modelo pre-entrenado usando un dataset específico y más pequeño, adaptando los pesos del modelo a una tarea o dominio particular. Incluye técnicas como SFT (Supervised Fine-Tuning), RLHF, DPO, y métodos parameter-efficient como LoRA y QLoRA.

⚙️ ¿Qué hace?

Ajusta los pesos del modelo (o un subconjunto con LoRA) para que "interiorice" un estilo, comportamiento, formato o conocimiento especializado. El resultado es un modelo que responde de forma distinta a nivel estructural, no solo por instrucción. Modifica la "personalidad base" del modelo.

✅ Ventajas
  • Estilo/tono consistente permanente
  • Menor uso de contexto en producción
  • Mejor en tareas muy específicas
  • LoRA/QLoRA reduce costo drásticamente
  • Puede ejecutarse en hardware propio
❌ Desventajas
  • Requiere datos curados de calidad
  • Riesgo de catastrofic forgetting
  • Caro y lento para modelos grandes
  • Difícil de debuggear
  • Actualización de conocimiento requiere re-training
🏗️ Variantes
  • Full Fine-Tuning
  • LoRA / QLoRA
  • RLHF (con feedback humano)
  • DPO (Direct Preference Optimization)
  • PEFT (Parameter Efficient)
  • Instruction Tuning
🟢 Cuándo USARLO
  • Estilo/voz de marca muy específico
  • Formato de output rígido y consistente
  • Dominio muy técnico (medicina, derecho, código)
  • Prompt engineering ya no da más resultados
  • Reducir costos de inferencia a largo plazo
🔴 Cuándo NO usarlo
  • Menos de 500-1000 ejemplos de calidad
  • El conocimiento cambia frecuentemente
  • Budget limitado (usar LoRA si no hay opción)
  • RAG o prompts resuelven el problema

04
In-Context Learning (ICL) & Long Context
Enseñar sin entrenar, todo dentro del prompt
$$ Medio Fácil
▾
📖 ¿Qué es?

Capacidad emergente de los LLMs de aprender de ejemplos dados directamente en el contexto del prompt (few-shot). Con ventanas de contexto de 1M+ tokens (Gemini 2.0, Claude 3.5), es posible meter libros enteros, codebases completos o datasets completos como "contexto vivo" sin necesidad de fine-tuning.

⚙️ ¿Qué hace?

El modelo infiere el patrón deseado a partir de los ejemplos dados (1-shot, few-shot, many-shot). Con long context, permite "cargar" documentación completa, historiales, código fuente, o bases de conocimiento directamente en memoria temporal del modelo durante la sesión.

✅ Ventajas
  • Sin modificar el modelo base
  • Flexible y actualizable instantáneamente
  • Ideal para muchos ejemplos heterogéneos
  • Ventanas largas eliminan necesidad de RAG en muchos casos
❌ Desventajas
  • Costo de tokens muy alto con contextos largos
  • Latencia alta (prefill cost)
  • Lost-in-the-middle: el modelo olvida el centro
  • No persiste entre conversaciones
📏 Límites de contexto 2026
  • Gemini 2.0: 2M tokens
  • Claude 3.7: 200K tokens
  • GPT-4o: 128K tokens
  • Llama 3.3 70B: 128K tokens
  • Mistral Large: 128K tokens
🟢 Cuándo USARLO
  • Prototipos donde no hay tiempo para RAG/FT
  • Análisis de documentos largos (contratos, research)
  • Many-shot learning para nuevas tareas
  • Cuando los datos son muy dinámicos
🔴 Cuándo NO usarlo
  • Producción con millones de queries (muy caro)
  • Necesitás recuperación precisa de un hecho específico
  • Datos que no caben en ningún contexto
🔬 Estado del Arte 2026

Las ventanas de contexto gigantes están redefiniendo el panorama. Gemini 2.0 con 2M tokens puede procesar toda una codebase o una novela completa. Sin embargo, estudios como "Lost in the Middle" confirman que los modelos pierden atención en el centro del contexto. La técnica Many-Shot ICL (cientos de ejemplos en el prompt) ha demostrado igualar fine-tuning en muchos benchmarks, democratizando el ajuste sin entrenamiento.

05
Agentes y Tool Use (Function Calling)
Dale al modelo manos, ojos y capacidad de actuar
$$ Medio Intermedio
▾
📖 ¿Qué es?

Extensión del LLM mediante herramientas externas que el modelo puede "llamar": APIs, bases de datos, calculadoras, navegadores web, código Python, etc. El modelo decide cuándo y cómo usar cada herramienta (ReAct loop). Los agentes multi-step pueden planificar, actuar, observar y re-planificar autónomamente.

⚙️ ¿Qué hace?

Transforma el LLM de un sistema de Q&A a un sistema que puede ejecutar acciones reales: enviar emails, consultar bases de datos en tiempo real, ejecutar código, navegar por la web, interactuar con sistemas externos (CRMs, ERPs, APIs). Arquitecturas multi-agente permiten dividir tareas complejas entre agentes especializados.

✅ Ventajas
  • Capacidades ilimitadas via herramientas
  • Acceso a datos en tiempo real
  • Puede automatizar workflows complejos
  • Modular y extensible
  • Combina con RAG, FT y otros métodos
❌ Desventajas
  • Latencia alta (múltiples LLM calls)
  • Costo elevado por cadena de llamadas
  • Difícil de debuggear y monitorear
  • Riesgo de loops o acciones no deseadas
  • Requiere buena orquestación
🏗️ Patrones comunes
  • ReAct (Reason + Act)
  • Plan & Execute
  • Multi-Agent Systems
  • Reflexion (auto-crítica)
  • Computer Use (GUI agents)
  • MCP (Model Context Protocol)
🟢 Cuándo USARLO
  • Automatización de procesos multi-paso
  • Integración con sistemas externos
  • Tareas que requieren información en tiempo real
  • Workflows que mezclan datos y acciones
  • Asistentes con capacidad de "hacer"
🔴 Cuándo NO usarlo
  • Respuestas simples sin acciones externas
  • Latencia ultra-baja (<100ms)
  • Entornos sin confianza para ejecución de código
  • Sistemas sin APIs accesibles

06
Knowledge Distillation
Comprimir el conocimiento de un modelo grande en uno pequeño
$$$ Alto Avanzado
▾
📖 ¿Qué es?

Técnica para transferir el "conocimiento" de un modelo grande (teacher) a un modelo más pequeño (student). El modelo student aprende a imitar las distribuciones de probabilidad del teacher, no solo las respuestas correctas, logrando un modelo más pequeño con rendimiento cercano al grande.

⚙️ ¿Qué hace?

El modelo teacher genera "soft labels" (probabilidades suavizadas) que el modelo student usa para aprender. Hay dos variantes: offline distillation (generar dataset con el teacher primero) y online distillation (aprendizaje simultáneo). Se usa para crear modelos on-device, edge computing, o reducir costos de inferencia.

✅ Ventajas
  • Modelos mucho más pequeños y rápidos
  • Menor costo de inferencia
  • Deployable en edge / on-device
  • Mantiene capacidades clave del modelo grande
❌ Desventajas
  • Proceso complejo y costoso de setup
  • Siempre habrá pérdida de capacidad
  • Requiere acceso a model weights del teacher
  • Difícil de hacer con modelos propietarios
🎯 Casos reales
  • Phi-3 (MSFT): destilado de GPT-4
  • DistilBERT: destilado de BERT
  • TinyLlama
  • Apple On-Device LLMs
  • Gemma (Google)
🟢 Cuándo USARLO
  • Deployment en mobile / edge
  • Reducir costos de inferencia en producción masiva
  • Latencia muy baja requerida
  • Privacidad: datos no pueden salir del dispositivo
🔴 Cuándo NO usarlo
  • Tareas que requieren el máximo razonamiento
  • Sin acceso a compute significativo para el proceso
  • Aplicación de bajo volumen donde el costo no importa
🔬 Estado del Arte 2026

La família Phi de Microsoft es el ejemplo más exitoso: Phi-3-mini (3.8B) rival a modelos 10x más grandes. Apple MLX y los modelos on-device de iOS 18 utilizan destilación masiva. La tendencia 2026 es generar datos sintéticos de alta calidad con modelos frontier (GPT-4, Claude) para entrenar modelos pequeños muy capaces — técnica llamada "self-play distillation" o Synthetic Data Fine-Tuning.

07
Guardrails, System Prompts & Constitutional AI
Controlar y alinear el comportamiento del modelo
$ Bajo Intermedio
▾
📖 ¿Qué es?

Conjunto de técnicas para controlar, limitar y alinear el comportamiento del modelo: system prompts de configuración, filtros de input/output, validadores de formato, modelos de clasificación para detectar contenido inapropiado, y técnicas de RLHF/RLAIF para alinear valores del modelo con los deseados.

⚙️ ¿Qué hace?

Define la "personalidad" y límites del modelo mediante instrucciones persistentes (system prompt), agrega capas de validación que interceptan inputs/outputs antes y después del LLM, y en el caso de Constitutional AI, usa el propio modelo para auto-criticar y corregir sus respuestas según principios definidos.

✅ Ventajas
  • Control granular sin re-entrenar
  • Rápido de implementar
  • Crítico para compliance y seguridad
  • Combina con cualquier otro método
❌ Desventajas
  • Vulnerable a prompt injection
  • Añade latencia si hay múltiples capas
  • Puede ser demasiado restrictivo
  • System prompts consumen contexto
🛡️ Tipos
  • Input filters (pre-LLM)
  • Output validators (post-LLM)
  • System prompts
  • Constitutional AI (Anthropic)
  • RLHF / RLAIF
  • Structured outputs (JSON mode)
🟢 Cuándo USARLO
  • Siempre en producción (mínimo system prompt)
  • Apps con usuarios finales (B2C)
  • Industrias reguladas (salud, finanzas, legal)
  • Cuando el formato de output es crítico
🔴 Cuándo NO omitirlo
  • Nunca omitir en producción con usuarios reales
  • Especialmente crítico en LLMs sin fine-tuning de alineamiento

08
Domain-Adaptive Pretraining (DAPT) & Continued Pretraining
Construir un modelo que "habla" tu dominio desde los cimientos
$$$$+ Muy Alto Experto
▾
📖 ¿Qué es?

Continuar el pre-entrenamiento de un modelo base con grandes volúmenes de texto de un dominio específico (medicina, derecho, código, idioma regional, etc.) antes de cualquier instruction tuning. Es la base sobre la que se construyen modelos como BioMedLM, LegalBERT, o CodeLlama. Requiere cientos de GB a TB de datos y semanas/meses de compute.

⚙️ ¿Qué hace?

Actualiza los pesos del transformer en la tarea de next-token prediction usando corpus especializados masivos. Permite al modelo internalizar vocabulario, sintaxis, conceptos y relaciones propias del dominio que no estaban suficientemente representadas en los datos de pretraining general. El resultado es un modelo con intuición profunda del dominio.

✅ Ventajas
  • Comprensión profunda del dominio
  • Base sólida para fine-tuning posterior
  • Vocabulario especializado nativo
  • Puede aprender idiomas infrarepresentados
❌ Desventajas
  • Costo computacional enorme (GPU weeks/months)
  • Requiere datasets masivos y limpios
  • Solo para organizaciones grandes
  • Riesgo de perder capacidades generales
  • ROI difícil de justificar para la mayoría
🎯 Modelos reales
  • CodeLlama (código)
  • BioMedLM (biomedicina)
  • LegalBERT (derecho)
  • BloombergGPT (finanzas)
  • StarCoder (código)
  • Modelos de idiomas indígenas
🟢 Cuándo USARLO
  • Dominio altamente especializado con terminología única
  • Idioma no cubierto adecuadamente en modelos generales
  • Tienes acceso a corpus masivo propio (TB de datos)
  • Producto de nicho que justifica el costo
🔴 Cuándo NO usarlo
  • La mayoría de los casos empresariales
  • RAG + Fine-tuning resuelve el problema
  • Sin presupuesto de cómputo significativo
  • Sin expertise en training LLMs a escala
🔬 Estado del Arte 2026

Con modelos de propósito general cada vez más capaces, el DAPT se justifica principalmente para dominios muy especializados o idiomas infrarepresentados. El caso de BloombergGPT para finanzas sigue siendo el benchmark de referencia. La tendencia 2026 es Continued Pretraining eficiente usando técnicas como LoRA aplicado al pretraining, reduciendo costos. Para la mayoría de casos, la combinación RAG + Fine-tuning supera al DAPT en ROI.

¿Cuál usar? Árbol de Decisión

¿Empezando / prototipando?

→ Prompt Engineering primero. Siempre. Agota este método antes de invertir en los otros.

¿Necesitás conocimiento propio actualizable?

→ RAG. Es la solución más versátil para conocimiento dinámico o propietario.

¿Querés cambiar el estilo/comportamiento base?

→ Fine-Tuning (LoRA). Especialmente si el prompt engineering no da el resultado deseado.

¿Necesitás que el modelo "actúe" en el mundo?

→ Agentes + Tool Use. Con APIs, bases de datos, acciones externas.

¿Deployment en edge/mobile con bajo costo?

→ Knowledge Distillation. O usar modelos ya destilados (Phi-3, Gemma).

¿Dominio ultra-especializado sin modelo existente?

→ DAPT. Solo si tenés TBs de datos y presupuesto de cómputo serio.

💡 Regla de oro 2026: La mayoría de casos enterprise se resuelven con la combinación RAG + Fine-Tuning (LoRA) + Guardrails. Los agentes se agregan cuando se necesita automatización. El prompt engineering nunca desaparece: siempre es la primera capa. DAPT y Distilación son para casos muy específicos o a escala masiva.
