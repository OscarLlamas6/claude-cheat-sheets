Visión General

Arquitecturas RAG y estrategias de chunking — contexto, relaciones y por qué importan

¿Qué es RAG? Retrieval-Augmented Generation combina un motor de recuperación (retriever) con un modelo generativo (LLM). La calidad final depende de tres factores: calidad del índice, estrategia de chunking y arquitectura RAG. Cambiar cualquiera de los tres afecta el resultado final de forma dramática.
📦

Chunking

Cómo divides tus documentos en piezas recuperables. Es la base de todo; un mal chunking no se puede compensar con un buen modelo. El chunking define el granularity-recall tradeoff: chunks pequeños = más precisión, chunks grandes = más contexto.

🔍

Retrieval

Cómo encuentras los chunks más relevantes para una query. Puede ser denso (vectores semánticos), disperso (BM25/TF-IDF), híbrido, o a través de grafos. La estrategia de retrieval determina la calidad del contexto que llega al LLM.

🧠

Generation

El LLM que sintetiza los chunks recuperados en una respuesta final. El contexto inyectado, el prompt engineering, el reranking previo y la estrategia de augmentation determinan cuánto "aprovecha" el modelo la información recuperada.

Tipos de RAG disponibles
🟢

Naive RAG

El pipeline básico. Documenta → chunk → embed → retrieve → generate. Funciona sorprendentemente bien para casos simples pero falla en queries complejas, multi-hop, o cuando los documentos tienen mala estructura.

🔧

Advanced RAG

Añade pre-retrieval (rewrite de query, HyDE), retrieval optimizado (reranking, hybrid search) y post-retrieval (compresión de contexto, filtros). La evolución natural del naive RAG para producción.

🧩

Modular RAG

Diseño plug-and-play donde cada componente (router, retriever, reranker, generator) es intercambiable. Máxima flexibilidad pero mayor complejidad arquitectónica. El estándar de la industria en 2025.

🕸️

Graph RAG

Construye un knowledge graph sobre los documentos. Excelente para razonamiento multi-hop y relaciones complejas. Desarrollado por Microsoft Research. Costoso en indexado pero poderoso en consulta.

🤖

Agentic RAG

Un agente LLM decide dinámicamente cuándo recuperar, qué recuperar, si los resultados son suficientes y si debe iterar. La frontera actual del estado del arte para tareas complejas y open-ended.

🔄

Self-RAG / CRAG

El modelo evalúa la relevancia de los documentos recuperados y decide si usarlos, ignorarlos o buscar más. Reduce alucinaciones causadas por retrieval de baja calidad.

Filtrar:
📄
Naive RAG
Complejidad: Baja Básico

El pipeline original y más simple. Chunk → embed → store → retrieve → generate. Es el punto de partida de todo RAG.

¿Qué hace?
Divide documentos en chunks, los embeds en un vector store, y en tiempo de query recupera los top-K chunks más similares semánticamente y los pasa al LLM como contexto.
¿Qué NO hace?
No maneja queries complejas o multi-hop. No tiene mecanismos de corrección. No evalúa la calidad de lo recuperado. No itera. No reescribe queries.
Ventajas
  • Implementación trivial
  • Bajo costo operativo
  • Fácil de debuggear
  • Baseline sólido para comparar
Desventajas
  • Muy sensible al chunking
  • Falla en razonamiento multi-hop
  • Sin corrección de errores de retrieval
  • No maneja ambigüedad de queries
✓ Úsalo cuando

POC, demos, datasets pequeños y bien estructurados, queries simples y directas, presupuesto/tiempo limitado.

✗ No lo uses cuando

Queries complejas, documentos técnicos densos, necesitas alta precisión en producción, o los usuarios esperan razonamiento multi-step.

Mejor para → POCs, prototipos rápidos, FAQs simples
⚡
Advanced RAG
Complejidad: Media Avanzado

Optimizaciones en las tres fases: pre-retrieval, retrieval y post-retrieval. La evolución natural del Naive RAG para entornos de producción.

¿Qué hace?
Pre-retrieval: reescritura de query, HyDE, query decomposition. Retrieval: hybrid search (sparse + dense), reranking (cross-encoders). Post-retrieval: compresión de contexto, reordenamiento por relevancia, filtros de calidad.
¿Qué NO hace?
No decide autónomamente cuándo recuperar. No itera en múltiples rondas. No construye conocimiento estructurado relacional. Es aún un pipeline lineal pero bien optimizado.
Ventajas
  • Mejora dramática sobre Naive
  • Bien documentado y con tooling maduro
  • Componentes combinables a la carta
  • Buen balance complejidad/resultado
Desventajas
  • Mayor latencia que Naive
  • Más componentes = más puntos de falla
  • Aún lineal, sin autonomía
  • Reranking puede ser costoso
✓ Úsalo cuando

Producción con usuarios reales, documentos técnicos, queries variadas, cuando el Naive RAG no alcanza la precisión requerida.

✗ No lo uses cuando

La latencia es crítica (sub-100ms) o cuando el dataset es pequeño y simple donde el overhead no vale la pena.

Mejor para → Producción enterprise, chatbots de soporte técnico
🧩
Modular RAG
Complejidad: Alta Arquitectura

Framework donde cada componente del pipeline (router, retriever, reranker, memory, generator) es un módulo independiente intercambiable. El estándar en sistemas serios.

¿Qué hace?
Permite construir pipelines complejos combinando módulos: un Search Module (web, vector, graph), un Memory Module (short/long-term), un Fusion Module para combinar múltiples fuentes, y un Routing Module que decide qué pipeline usar según el tipo de query.
¿Qué NO hace?
No simplifica el sistema — lo hace más poderoso a costa de mayor complejidad de orquestación. No toma decisiones autónomas como un agente.
Ventajas
  • Máxima flexibilidad y extensibilidad
  • Componentes testeables aisladamente
  • Fácil A/B testing de módulos
  • Escala bien en equipos grandes
Desventajas
  • Overhead de diseño y mantenimiento
  • Debugging más complejo
  • Requiere buena infraestructura
  • Overkill para proyectos pequeños
✓ Úsalo cuando

Múltiples fuentes de datos heterogéneas, equipos que evolucionan el sistema constantemente, necesitas diferentes estrategias para diferentes tipos de queries.

✗ No lo uses cuando

Equipo pequeño, MVP, caso de uso único y bien definido donde la modularidad no aporta valor real.

Mejor para → Plataformas enterprise multi-dominio, productos maduros
🕸️
Graph RAG
Complejidad: Alta Especializado Microsoft Research

Construye un knowledge graph (entidades + relaciones) durante la indexación. Recupera por recorrido de grafo en lugar de similitud vectorial. Game-changer para razonamiento relacional.

¿Qué hace?
Durante indexación: extrae entidades, relaciones y comunidades del corpus con un LLM, construyendo un grafo. En retrieval: traversal del grafo para encontrar información relacionada multi-hop que la similitud coseno nunca encontraría. Soporta preguntas "globales" sobre el corpus completo.
¿Qué NO hace?
No es bueno para búsquedas simples por similitud directa. No funciona bien con datos no relacionales o muy heterogéneos. El costo de indexación puede ser 10-100x mayor que RAG tradicional.
Ventajas
  • Razonamiento multi-hop nativo
  • Queries globales sobre todo el corpus
  • Explícita las relaciones entre entidades
  • Reduce alucinaciones en dominios densos
Desventajas
  • Indexación muy costosa (tokens LLM)
  • Requiere grafos bien mantenidos
  • Latencia alta en retrieval
  • Difícil de implementar correctamente
✓ Úsalo cuando

Dominios con relaciones complejas (legal, médico, científico), preguntas como "¿qué tiene en común X e Y?", análisis de redes de entidades.

✗ No lo uses cuando

Documentos sin estructura relacional clara, presupuesto de indexación ajustado, updates frecuentes del corpus (el grafo se desactualiza).

Mejor para → Análisis legal, knowledge bases científicas, inteligencia competitiva
💡
HyDE — Hypothetical Document Embeddings
Complejidad: Media Técnica Pre-Retrieval

En lugar de embeddear la query directamente, el LLM genera un documento hipotético que respondería la pregunta. Ese documento hipotético se usa para el retrieval. Reduce el gap semántico entre queries y documentos.

¿Qué hace?
Query → LLM genera respuesta hipotética (aunque sea incorrecta en hechos, el estilo semántico es el correcto) → embed la respuesta hipotética → buscar chunks similares a esa respuesta hipotética. El embedding de una respuesta "suena" más a los documentos reales que el embedding de una pregunta.
¿Qué NO hace?
No resuelve el problema si el LLM genera hipotéticos muy off-domain. Añade una llamada extra al LLM. No funciona bien con queries factuales muy específicas donde no hay "espacio" para la hipótesis.
Ventajas
  • Mejora recall en queries abstractas
  • Reduce mismatch query-documento
  • Fácil de añadir a un pipeline existente
  • Muy efectivo con documentos técnicos
Desventajas
  • Latencia extra (1 LLM call adicional)
  • Costo adicional por token
  • Si el hipotético es muy malo, degrada el resultado
✓ Úsalo cuando

Queries conceptuales o abstractas, documentos técnicos muy densos, embeddings dan retrieval de baja calidad, queries en estilo diferente al de los documentos.

✗ No lo uses cuando

Latencia crítica, queries muy específicas y factuales, presupuesto ajustado de tokens.

Mejor para → Documentación técnica, papers científicos, bases de conocimiento especializadas
🪞
Self-RAG
Complejidad: Alta Self-Reflective

El modelo decide dinámicamente cuándo recuperar, evalúa la relevancia de cada chunk recuperado, y critica su propia respuesta generada antes de entregarla. Introduce "reflection tokens" para control fino.

¿Qué hace?
Usa tokens especiales: [Retrieve] para decidir si recuperar, [IsRel] para evaluar si el chunk es relevante, [IsSup] para verificar si la respuesta está soportada por el contexto, [IsUse] para evaluar la utilidad de la respuesta final. Requiere fine-tuning del LLM base.
¿Qué NO hace?
No funciona out-of-the-box con modelos genéricos — requiere fine-tuning o un LLM con capacidades de instrucción fuertes. No es un drop-in replacement simple.
Ventajas
  • Reduce alucinaciones significativamente
  • Retrieval bajo demanda (eficiente)
  • Autocontrol de calidad integrado
  • Estado del arte en factualidad
Desventajas
  • Requiere modelo fine-tuneado
  • Complejo de implementar correctamente
  • Mayor latencia por rondas de evaluación
  • Tooling limitado vs Advanced RAG
✓ Úsalo cuando

Factualidad crítica (médico, legal, financiero), cuando las alucinaciones tienen consecuencias reales, tienes recursos para fine-tuning.

✗ No lo uses cuando

No puedes hacer fine-tuning, necesitas respuestas rápidas, o el dominio no justifica la complejidad adicional.

Mejor para → Aplicaciones médicas, legales, financieras donde factualidad es crítica
🔧
CRAG — Corrective RAG
Complejidad: Media Self-Corrective

Evalúa la calidad de los documentos recuperados y, si son de baja confianza, activa una búsqueda web como fallback o reformula la query. Añade un "corrective step" al pipeline estándar.

¿Qué hace?
Después del retrieval, un evaluador (LLM ligero) clasifica los documentos en: Correcto (alta confianza → usar directamente), Incorrecto (baja confianza → descartar, usar web search), Ambiguo (confianza media → usar con cautela + refinar). Puede integrar Tavily/SerpAPI para búsqueda web como corrección.
¿Qué NO hace?
No aprende de los errores para mejorar el índice. No hace múltiples rondas de búsqueda. La calidad del evaluador limita la calidad de la corrección.
Ventajas
  • Degradación elegante ante retrieval pobre
  • No requiere fine-tuning
  • Fácil de implementar con LangGraph
  • Integrable con búsqueda web
Desventajas
  • El evaluador puede equivocarse
  • Web search añade latencia y costo
  • Dos LLM calls mínimo por query
✓ Úsalo cuando

Tu índice tiene gaps de conocimiento, queries fuera del dominio son posibles, necesitas un fallback a información actualizada vía web.

✗ No lo uses cuando

Datos completamente offline (sin acceso a web), latencia muy crítica, dominio perfectamente cubierto por el índice.

Mejor para → Asistentes que combinan knowledge base propia + información actualizada
🤖
Agentic RAG
Complejidad: Muy Alta Agéntico Estado del arte 2025

Un agente LLM con acceso a herramientas de retrieval como tools. Decide autónomamente cuándo buscar, qué buscar, si los resultados son suficientes, y si debe iterar. El paradigma dominante en 2025-2026.

¿Qué hace?
El LLM tiene tools: search_knowledge_base(), web_search(), query_database(), read_document(). El agente planifica, ejecuta búsquedas iterativas, razona sobre los resultados, y puede hacer seguimiento multi-turn. Compatible con arquitecturas ReAct, CoT, y frameworks como LangGraph, AutoGen, CrewAI.
¿Qué NO hace?
No garantiza comportamiento determinista. Puede ejecutar demasiadas búsquedas (token waste). Sin buena observabilidad es una caja negra difícil de debuggear. Costoso si el agente no tiene buenos stop conditions.
Ventajas
  • Maneja queries abiertos y complejos
  • Adaptación dinámica al problema
  • Multi-source nativo
  • Puede planificar y descomponer tareas
Desventajas
  • Latencia alta e impredecible
  • Costo difícil de controlar
  • Debugging complejo
  • Riesgo de loops o stuck states
✓ Úsalo cuando

Research tasks complejos, análisis multi-documento, cuando el usuario necesita respuestas que requieren múltiples búsquedas y síntesis.

✗ No lo uses cuando

Necesitas latencia predecible, preguntas simples y directas, presupuesto de tokens ajustado, entornos con políticas de seguridad estrictas.

Mejor para → Asistentes de research, análisis de contratos, due diligence
🔀
RAG-Fusion
Complejidad: Media Multi-Query

Genera múltiples variaciones de la query, realiza retrieval para cada una, y fusiona los resultados con Reciprocal Rank Fusion (RRF). Combate el problema de que una sola query puede no capturar toda la intención.

¿Qué hace?
Query original → LLM genera N variaciones → retrieval paralelo para cada variación → RRF re-rankea todos los resultados combinados → top-K fusionados pasan al LLM. Reduce el sesgo de formulación de la query original y aumenta el recall.
¿Qué NO hace?
No mejora la calidad del índice base. Las variaciones de query generadas por el LLM pueden ser semánticamente similares entre sí (redundantes). El RRF no entiende semántica, solo rankings.
Ventajas
  • Mayor recall que single-query
  • Robusto ante formulaciones ambiguas
  • RRF es simple y efectivo
  • Fácil de implementar
Desventajas
  • N × latencia de retrieval
  • N LLM calls para generar variaciones
  • Variaciones pueden ser redundantes
✓ Úsalo cuando

Queries con múltiples interpretaciones posibles, usuarios con vocabulario variado, dominios con sinónimos o jerga mixta.

✗ No lo uses cuando

Queries muy específicas y factuales donde las variaciones no aportan, presupuesto de latencia/tokens ajustado.

Mejor para → Search engines internos, portales de conocimiento con usuarios diversos
🔁
Iterative / Multi-Hop RAG
Complejidad: Alta Multi-Step

Múltiples rondas de retrieval donde cada ronda usa el contexto acumulado de la anterior para refinar la siguiente búsqueda. Esencial para preguntas multi-hop que requieren encadenar información.

¿Qué hace?
Query → retrieval ronda 1 → LLM razona sobre los resultados → genera sub-query basada en gaps detectados → retrieval ronda 2 → ... → síntesis final. Útil para preguntas como "¿Qué relación existe entre X e Y?" donde la respuesta requiere conectar múltiples documentos.
¿Qué NO hace?
No garantiza que siempre converja. Sin buenos stop conditions puede iterar indefinidamente. No apto para tiempo real.
Ventajas
  • Razonamiento multi-hop genuino
  • Acumula contexto progresivamente
  • Adaptable a la complejidad de la pregunta
Desventajas
  • Latencia acumulada por rondas
  • Context window crece rápidamente
  • Difícil de predecir costo
✓ Úsalo cuando

Análisis profundo, preguntas comparativas complejas, investigación donde la respuesta requiere conectar múltiples fuentes no obvias.

✗ No lo uses cuando

Necesitas respuestas en segundos, preguntas simples de lookup, latencia predecible es requerimiento.

Mejor para → Análisis de investigación, due diligence, comparativas técnicas profundas
🎯
Adaptive RAG
Complejidad: Media-Alta Adaptive 2024

Clasifica la complejidad de la query y selecciona dinámicamente la estrategia de retrieval apropiada: sin retrieval (LLM solo), single-step, o multi-step. Optimiza costo/latencia/calidad automáticamente.

¿Qué hace?
Un clasificador (fine-tuned o LLM) evalúa la query y decide: Simple → el LLM responde solo (sin retrieval), Media → single-step RAG estándar, Compleja → multi-step iterative RAG. Evita el overhead de retrieval en queries que el LLM ya sabe responder.
¿Qué NO hace?
El clasificador puede equivocarse. Si clasifica como "simple" una query que requería retrieval, la respuesta puede alucinarse. Requiere calibración cuidadosa del umbral de clasificación.
Ventajas
  • Optimiza costo dinámicamente
  • Latencia baja para queries simples
  • Mejor experiencia de usuario
  • Uso eficiente de tokens
Desventajas
  • Riesgo de clasificación errónea
  • Necesita calibración del clasificador
  • Más componentes en el sistema
✓ Úsalo cuando

Mix de queries simples y complejas, optimización de costos es importante, los usuarios hacen desde preguntas triviales hasta análisis profundos.

✗ No lo uses cuando

Todas las queries son igualmente complejas, o cuando el riesgo de omitir retrieval en queries que lo necesitan es inaceptable.

Mejor para → Asistentes de uso general con queries muy variadas en complejidad
🖼️
Multimodal RAG
Complejidad: Alta Especializado 2024-2025

Extiende el RAG para indexar y recuperar no solo texto sino imágenes, tablas, gráficas, audio y vídeo. Fundamental cuando los documentos mezclan múltiples modalidades (PDFs con figuras, presentaciones, etc.).

¿Qué hace?
Estrategia 1: extrae texto de imágenes/tablas con OCR/VLM y lo indexa como texto. Estrategia 2: genera summaries de imágenes con un VLM y los indexa. Estrategia 3: usa embeddings multimodales (CLIP, ImageBind) para buscar por similitud cross-modal. El retrieval puede retornar imágenes, texto, o ambos al LLM multimodal.
¿Qué NO hace?
Sin VLM o embeddings multimodales, las imágenes son completamente invisibles. El costo de procesamiento de imágenes es significativamente mayor que texto puro.
Ventajas
  • Procesa PDFs reales (figuras, tablas)
  • Información en imágenes no se pierde
  • Necesario para documentación técnica
  • Mejor cobertura del corpus real
Desventajas
  • Alto costo de indexación
  • Requiere VLM (cara) o pipeline OCR
  • Embeddings multimodales menos maduros
✓ Úsalo cuando

Tus documentos tienen figuras relevantes (manuales técnicos, papers científicos, presentaciones), tablas complejas, diagramas con información clave.

✗ No lo uses cuando

Documentos puramente textuales, presupuesto ajustado, las imágenes no aportan información que no esté en el texto.

Mejor para → Manuales técnicos, reportes financieros, documentación con diagramas
📜
Long Context RAG
Complejidad: Media Especializado Post-2024

Aprovecha los context windows largos de modelos modernos (200k-1M tokens) para pasar documentos completos o grandes porciones al LLM, reduciendo o eliminando el paso de chunking/retrieval selectivo.

¿Qué hace?
En lugar de recuperar chunks específicos, carga documentos enteros o secciones largas en el contexto. Técnicas como Contextual Retrieval (Anthropic) añaden resúmenes de documento a cada chunk antes de indexar. Reduce la pérdida de contexto por fragmentación. Compatible con modelos Claude, Gemini 1.5+, GPT-4o.
¿Qué NO hace?
No soluciona el "lost in the middle" problem (los LLMs atienden mejor al principio y fin del contexto). Costo por token se dispara con contextos muy largos. No escala a millones de documentos.
Ventajas
  • Sin pérdida de contexto por chunking
  • Simplifica el pipeline
  • Mejor coherencia en respuestas
  • Ideal para análisis de contratos largos
Desventajas
  • Costo de tokens muy alto
  • No escala a corpus grandes
  • "Lost in the middle" problem
  • Latencia alta por contextos largos
✓ Úsalo cuando

Pocos documentos muy largos (contratos, libros, especificaciones técnicas), análisis que requiere visión holística del documento completo.

✗ No lo uses cuando

Corpus con miles de documentos, presupuesto ajustado de tokens, queries que solo necesitan información puntual (no holística).

Mejor para → Análisis de contratos, auditoría de código, revisión de documentos extensos

Estrategias de Chunking

12 estrategias de fragmentación — el cimiento de todo pipeline RAG efectivo

Por qué el chunking es crítico: Un chunk ideal es semánticamente completo (contiene una idea completa), recuperable (el embedding lo representa bien en el espacio vectorial), y útil (da contexto suficiente al LLM para responder). El tamaño óptimo depende del dominio: documentos legales necesitan chunks más grandes que FAQs técnicas. Docling HybridChunker es actualmente uno de los mejores para PDFs complejos.
Filtrar:
📏
Fixed-Size Chunking
Complejidad: Mínima Simple

Divide el texto en chunks de N tokens/caracteres fijos, con o sin overlap. La estrategia más simple posible. Ignora completamente la estructura del texto.

¿Qué hace?
Corta el texto cada N tokens (típicamente 256-1024). Con overlap, el final de un chunk se repite al inicio del siguiente (ej: overlap de 20% para preservar continuidad). Completamente agnóstico al contenido.
¿Qué NO hace?
No respeta oraciones, párrafos, secciones ni ninguna unidad semántica. Puede cortar a mitad de una idea crítica. El overlap ayuda pero no soluciona el problema fundamental.
Ventajas
  • Trivial de implementar
  • Velocidad máxima
  • Chunks de tamaño predecible
  • Sin dependencias externas
Desventajas
  • Destruye estructura semántica
  • Chunks pueden ser incoherentes
  • Embeddings de baja calidad
  • Overlap crea redundancia en el índice
✓ Úsalo cuando

Texto plano sin estructura (transcripciones, logs), prototipado rápido donde la calidad es secundaria, volúmenes enormes donde la velocidad importa.

✗ No lo uses cuando

Documentos estructurados (PDFs, artículos, contratos), precisión de retrieval es importante, producción con usuarios reales.

Mejor para → Baseline/prototipado, texto plano sin estructura
🔤
Recursive Character Text Splitting
Complejidad: Baja Estándar LangChain default

Intenta dividir por separadores jerárquicos: primero párrafos (\n\n), luego saltos de línea (\n), luego espacios, luego caracteres. Respeta mejor la estructura natural que fixed-size. El default de LangChain.

¿Qué hace?
Aplica separadores en orden de preferencia: si el chunk resultante es demasiado grande al dividir por \n\n, prueba con \n, luego con " ", luego con "". Prioriza mantener párrafos completos y solo fragmenta si no hay otra opción. Produce chunks más coherentes que fixed-size.
¿Qué NO hace?
No entiende semántica real — solo conoce caracteres separadores. No sabe qué es una "idea completa". No maneja bien tablas, listas o código. No reconoce headers o secciones del documento.
Ventajas
  • Mucho mejor que fixed-size
  • Respeta estructura básica del texto
  • Una línea de código (LangChain)
  • Customizable con separadores propios
Desventajas
  • Aún no comprende semántica
  • Falla con texto sin párrafos claros
  • No ideal para PDFs o HTML
✓ Úsalo cuando

Texto en Markdown bien formateado, artículos de blog, README files, documentación con estructura clara de párrafos.

✗ No lo uses cuando

PDFs con layout complejo, documentos con tablas/listas/código, máxima calidad de retrieval es prioritaria.

Mejor para → Markdown, HTML limpio, documentación web scrapeada
🧠
Semantic Chunking
Complejidad: Media Semántico

Divide el texto donde cambia el significado semántico. Embeddea oraciones individuales y calcula la similitud coseno entre oraciones consecutivas — cuando la similitud cae significativamente, corta ahí.

¿Qué hace?
Tokeniza en oraciones → embeddea cada oración → calcula similitud entre oraciones adyacentes → detecta "quiebres semánticos" donde la similitud cae por debajo de un threshold → divide ahí. Produce chunks que representan ideas semánticamente cohesivas, no solo unidades de tamaño.
¿Qué NO hace?
No considera la estructura del documento (headers, secciones). Los chunks resultantes tienen tamaño muy variable. El costo de embedding de todas las oraciones es alto. El threshold de corte requiere calibración.
Ventajas
  • Chunks semánticamente cohesivos
  • Embeddings de alta calidad resultantes
  • Respeta el flujo de ideas natural
  • Adaptativo al contenido
Desventajas
  • Caro en tiempo de indexación
  • Chunks de tamaño muy variable
  • Threshold difícil de calibrar
  • Requiere modelo de embedding bueno
✓ Úsalo cuando

Textos narrativos, artículos largos, libros, documentos donde los temas cambian gradualmente. Máxima calidad semántica de chunks.

✗ No lo uses cuando

Documentos altamente estructurados (ya tienen su propia división lógica), presupuesto de indexación ajustado, corpus en tiempo real.

Mejor para → Textos narrativos largos, libros, artículos de investigación
📋
Document-Structure / Layout-Aware Chunking
Complejidad: Media Estructural

Utiliza la estructura lógica del documento (secciones, subsecciones, headers H1-H6, párrafos) como guía de chunking. Respeta las divisiones naturales del autor del documento.

¿Qué hace?
Parsea el documento como árbol jerárquico (título → secciones → subsecciones → párrafos). Cada sección o subsección se convierte en un chunk. Si una sección es demasiado larga, se subdivide. Puede incluir el breadcrumb jerárquico (H1 > H2 > H3) como metadata de cada chunk para contexto adicional.
¿Qué NO hace?
Depende completamente de que el documento tenga buena estructura. PDFs mal maquetados o documentos escaneados sin estructura son problemáticos. No maneja documentos sin headers.
Ventajas
  • Chunks alineados con la intención del autor
  • Metadata jerárquica rica
  • Retrieval contextualizado
  • Natural para documentación técnica
Desventajas
  • Requiere documentos bien estructurados
  • Chunks de tamaño muy variable
  • Parser de estructura puede fallar
✓ Úsalo cuando

Documentación técnica con headers, wikis, manuales, políticas empresariales, cualquier documento con estructura jerárquica explícita.

✗ No lo uses cuando

PDFs escaneados sin OCR, texto plano sin estructura, documentos con estructura inconsistente o mal formada.

Mejor para → Documentación técnica, wikis corporativas, manuales estructurados
🪟
Sliding Window / Overlapping Chunks
Complejidad: Baja Simple

Fixed-size chunking con overlap deliberado entre chunks consecutivos. La ventana deslizante garantiza que ninguna información en la frontera entre chunks se pierda completamente.

¿Qué hace?
Chunk 1: tokens 0-500. Chunk 2: tokens 400-900 (100 tokens de overlap). Chunk 3: tokens 800-1300. El overlap típico es 10-20% del tamaño del chunk. Garantiza que texto en los bordes aparece en al menos dos chunks, aumentando el recall.
¿Qué NO hace?
Infla el índice con datos duplicados. No resuelve el problema de coherencia semántica. Con overlap muy grande, los chunks se vuelven redundantes y añaden ruido al retrieval.
Ventajas
  • Mejora recall en fronteras de chunks
  • Trivial de implementar
  • Reduce pérdida por cortes arbitrarios
Desventajas
  • Índice más grande (más costo de embedding)
  • Duplicados pueden saturar el top-K
  • Aún no considera semántica
✓ Úsalo cuando

El texto importante suele estar distribuido sin estructura clara, como transcripciones de conversaciones, notas libres, texto corrido.

✗ No lo uses cuando

El índice tiene restricciones de tamaño, documentos con estructura clara donde el overlap añade ruido, queries muy específicas donde los duplicados son problemáticos.

Mejor para → Transcripciones, chat logs, texto corrido sin estructura
🌲
Small-to-Big / Parent-Child Chunking
Complejidad: Media Jerárquico

Indexa chunks pequeños para retrieval preciso, pero entrega al LLM el chunk padre (más grande) que los contiene. Combina la precisión de embeddings pequeños con el contexto de chunks grandes.

¿Qué hace?
Crea dos niveles: Child chunks (pequeños, ~128 tokens) que se embedean e indexan para retrieval preciso. Parent chunks (grandes, ~512-2048 tokens) que se almacenan pero no se indexan directamente. Cuando un child chunk es recuperado, se devuelve su parent al LLM para dar más contexto.
¿Qué NO hace?
Requiere gestión de dos niveles de almacenamiento. El parent puede ser tan grande que sature el context window si se recuperan múltiples. La relación parent-child depende del chunking inicial.
Ventajas
  • Alta precisión de retrieval (chunks pequeños)
  • Alto contexto para generación (chunks grandes)
  • Resuelve el precision/context tradeoff
  • Muy efectivo en benchmarks
Desventajas
  • Infraestructura más compleja (dos stores)
  • Mayor uso de memoria
  • Los parents pueden ser demasiado grandes
✓ Úsalo cuando

Necesitas alta precisión de retrieval Y respuestas con contexto amplio. Documentos técnicos largos donde las respuestas requieren sección completa.

✗ No lo uses cuando

La infraestructura no soporta almacenamiento dual, queries simples de lookup, documentos cortos donde el overhead no vale.

Mejor para → Documentación extensa, knowledge bases técnicas profundas
🏷️
Contextual Retrieval (Anthropic)
Complejidad: Media Contextual Anthropic 2024

Antes de embeddear cada chunk, un LLM genera una descripción contextual breve que sitúa ese chunk dentro del documento completo. El chunk + su contexto se embeddean juntos. Reduce fallos de retrieval por falta de contexto.

¿Qué hace?
Para cada chunk: prompt al LLM con el documento completo + el chunk → el LLM genera 2-3 oraciones de contexto situacional → se anteponen al chunk → se embeddea el chunk enriquecido. Ejemplo: "Este fragmento es del manual de instalación de X, sección de configuración de red, y describe los pasos para configurar la IP estática." Reduce fallos de retrieval en ~49% según Anthropic.
¿Qué NO hace?
Requiere pasar el documento completo al LLM para cada chunk → costo de tokens muy alto en indexación. No es práctico para corpus de millones de documentos sin optimización (prompt caching).
Ventajas
  • Mejora dramática del retrieval
  • Cada chunk tiene contexto propio
  • Reduce ambigüedad de chunks aislados
  • Compatible con cualquier base de chunking
Desventajas
  • Costo de indexación muy alto
  • Lento para corpus grandes
  • Necesita prompt caching para ser viable
✓ Úsalo cuando

Corpus de tamaño medio donde el costo extra de indexación es aceptable, documentos con chunks que pierden significado fuera de contexto, máxima calidad de retrieval.

✗ No lo uses cuando

Corpus masivos (millones de docs), updates frecuentes del corpus, presupuesto de indexación muy ajustado.

Mejor para → Knowledge bases medianas donde calidad > costo de indexación
⚙️
Docling HybridChunker
Complejidad: Media Estructural + Semántico IBM / Docling 2024

Combina chunking basado en la estructura del documento (layout parsing de PDF) con un tokenizador que respeta límites de tokens del modelo de embedding. Diseñado específicamente para PDFs complejos con tablas, figuras y listas.

¿Qué hace?
Docling parsea el PDF extrayendo su estructura semántica real (no solo texto plano): identifica títulos, párrafos, tablas, figuras, listas y captions. El HybridChunker respeta estas unidades estructurales y además asegura que los chunks no excedan el límite de tokens del tokenizador. Fusiona chunks pequeños adyacentes del mismo tipo si caben dentro del límite. Es layout-aware + tokenizer-aware simultáneamente.
¿Qué NO hace?
Requiere Docling como dependencia (parseo pesado de PDF). Más lento que chunkers simples en indexación. No añade contexto semántico al chunk (eso sería complementarlo con Contextual Retrieval).
Ventajas
  • Layout-aware nativo (tablas, figuras)
  • Chunks de tamaño controlado
  • Metadatos estructurales ricos
  • Mejor que cualquier chunker de texto puro para PDFs
  • Handles tablas como unidades coherentes
Desventajas
  • Solo funciona bien con PDFs (no web, MD)
  • Dependencia pesada (Docling)
  • Más lento en indexación
  • Requiere calibración del tokenizador
✓ Úsalo cuando

Tu corpus es principalmente PDFs con layout complejo (reportes, papers, manuales), documentos con tablas importantes, necesitas máxima fidelidad al documento original.

✗ No lo uses cuando

Corpus principalmente Markdown/HTML/texto plano, necesitas velocidad de indexación máxima, los PDFs son simples (solo texto corrido).

Mejor para → PDFs complejos con tablas/figuras — reportes, papers, manuales técnicos
🤖
Agentic Chunking (LLM-based)
Complejidad: Alta LLM-Driven

Un LLM analiza el documento y decide dónde colocar los límites de los chunks basándose en la semántica, el contexto y el propósito de cada fragmento. El chunking más "inteligente" posible.

¿Qué hace?
El LLM recibe bloques del documento y determina: ¿este fragmento es una idea completa? ¿Debe estar con el siguiente párrafo o separado? ¿Es una proposición independiente o parte de una argumentación que no debe fragmentarse? Puede generar chunks como proposiciones atómicas (Propositional Chunking) para máxima granularidad y precisión.
¿Qué NO hace?
Prohibitivamente caro para corpus grandes — cada documento requiere múltiples llamadas LLM. No determinista (el mismo documento puede chunkearse diferente en cada run). Latencia de indexación muy alta.
Ventajas
  • Máxima calidad semántica de chunks
  • Chunks como proposiciones atómicas
  • Adaptado completamente al contenido
  • Excelente para retrieval preciso
Desventajas
  • Costo de indexación extremadamente alto
  • Lentísimo para corpus grandes
  • No determinista
  • Difícil de escalar
✓ Úsalo cuando

Corpus pequeño y muy valioso donde la calidad es lo más importante, documentos complejos que ningún chunker automático maneja bien.

✗ No lo uses cuando

Corpus de miles de documentos, updates frecuentes, presupuesto ajustado de tokens, tiempo de indexación restringido.

Mejor para → Corpus pequeños de alta densidad informativa, knowledge bases premium
⏱️
Late Chunking
Complejidad: Media Contextual JinaAI 2024

Embeddea el documento completo primero (preservando el attention entre todas las partes) y luego extrae los embeddings de los chunks del output del modelo. Cada chunk tiene contexto del documento completo en su embedding.

¿Qué hace?
A diferencia del chunking tradicional que embeddea fragmentos aislados, Late Chunking pasa el documento completo por el encoder, obtiene los token embeddings con atención global, y luego hace mean pooling sobre los tokens que corresponden a cada chunk. El resultado: embeddings de chunks que "conocen" el contexto del documento completo.
¿Qué NO hace?
Limitado por el context window del modelo de embedding (aunque los modelos de JinaAI soportan hasta 8192 tokens). No funciona con documentos más largos que el context window del encoder.
Ventajas
  • Embeddings con contexto global del documento
  • Mejor representación de chunks ambiguos
  • Sin costo extra de LLM para contexto
  • Más coherente que embeddings aislados
Desventajas
  • Limitado al context window del encoder
  • Requiere modelos de embedding específicos
  • Más lento que embedding por chunks
✓ Úsalo cuando

Los chunks aislados son ambiguos sin contexto (texto técnico con muchas referencias cruzadas), documentos dentro del context window del encoder.

✗ No lo uses cuando

Documentos muy largos que exceden el context window del encoder, cuando los chunks son semánticamente independientes por naturaleza.

Mejor para → Documentos técnicos con referencias cruzadas internas, código fuente
🔭
Sentence Window Retrieval
Complejidad: Media Contextual

Indexa oraciones individuales para máxima precisión de embedding, pero al recuperar una oración, entrega al LLM una ventana de N oraciones alrededor de ella (contexto circundante). Similar a Parent-Child pero a nivel de oración.

¿Qué hace?
Cada oración se embeddea individualmente y se almacena con un puntero a su posición en el documento. Al recuperar la oración más relevante, el sistema expande la ventana: devuelve las K oraciones anteriores y posteriores para dar contexto al LLM. Maximiza la precisión de matching mientras mantiene contexto de lectura.
¿Qué NO hace?
El índice es muy grande (una entrada por oración). Las oraciones muy cortas tienen embeddings de baja calidad. La ventana de contexto puede incluir texto de otra sección si el documento no tiene separadores claros.
Ventajas
  • Alta precisión de retrieval
  • Contexto apropiado para el LLM
  • Bueno para búsquedas muy específicas
Desventajas
  • Índice masivo (muchas oraciones)
  • Oraciones cortas = embeddings pobres
  • Ventana puede cruzar secciones
✓ Úsalo cuando

Queries muy precisas y específicas, Q&A sobre hechos puntuales, texto narrativo denso con alta densidad informativa.

✗ No lo uses cuando

Los documentos tienen oraciones muy cortas o fragmentadas, presupuesto de almacenamiento ajustado, queries conceptuales amplias.

Mejor para → Bases médicas, bases legales, Q&A sobre hechos específicos
🦖
RAPTOR — Recursive Abstractive Processing
Complejidad: Alta Jerárquico Stanford 2024

Construye un árbol jerárquico de resúmenes: los chunks base se resumen en clusters, esos resúmenes se resumen nuevamente, creando niveles de abstracción. Permite queries tanto detalladas como de alto nivel.

¿Qué hace?
Nivel 0: chunks base. Nivel 1: clusters de chunks similares → resumen por LLM. Nivel 2: clusters de resúmenes → resumen de resúmenes. Y así recursivamente. El índice final contiene todos los niveles. En retrieval, puede recuperar desde el nivel más detallado hasta el más abstracto según lo requiera la query.
¿Qué NO hace?
Muy costoso en indexación (múltiples rondas de LLM para resúmenes). Difícil de mantener actualizado (cualquier cambio requiere re-construir el árbol). Complejidad de implementación alta.
Ventajas
  • Maneja queries tanto locales como globales
  • Comprensión multi-granularidad
  • Excelente para corpus con jerarquía natural
  • State of the art en varios benchmarks
Desventajas
  • Costo de indexación muy alto
  • Actualizaciones del corpus caras
  • Implementación compleja
  • Difícil de debuggear
✓ Úsalo cuando

Corpus estático que no cambia frecuentemente, necesitas tanto detail-seeking queries como high-level summary queries, corpus con estructura jerárquica natural (libros, reportes anuales).

✗ No lo uses cuando

Corpus con updates frecuentes, presupuesto de indexación ajustado, sistema que necesita simplicidad de mantenimiento.

Mejor para → Libros, reportes extensos, corpus estático con jerarquía natural

Recomendaciones para tu caso (empresa)
Estás usando Docling HybridChunker — buena elección si tu corpus son PDFs. Para escalar a producción te recomendaría el siguiente stack: Docling HybridChunker + Contextual Retrieval (añade contexto a cada chunk con Claude) + Hybrid Search (BM25 + dense embeddings) + Cross-encoder reranking (cohere-rerank o similar) + arquitectura Advanced RAG / Modular RAG. Si tus queries son complejas y multi-hop, considera añadir RAG-Fusion en la capa de pre-retrieval. Para el futuro próximo: evalúa Agentic RAG con LangGraph para flujos de análisis profundo. El HybridChunker es excelente para PDFs — complementarlo con Contextual Retrieval es el next step de mayor ROI.
